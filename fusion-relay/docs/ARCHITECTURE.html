设计目标

核心思路：参考 agentfw 的 Wire Tap 反向代理架构和 OpenRouter Fusion 的 Panel→Judge→Synthesizer 流水线，构建一个本地可控、OpenAI 兼容的 API 中转站，让下游 Agent / 应用感知不到多模型协作，只看到"一个更聪明的模型"。

3. 双协议入站设计

🎯 关键洞察：零协议转换

5 家头部国产模型（DeepSeek / MiniMax / GLM / Kimi / Qwen）均原生支持 OpenAI + Anthropic 双协议——每家都有独立的 Anthropic 兼容端点。

这意味着：Fusion 中转站的出站方向完全不需要协议转换——所有模型都能直接接受 OpenAI 格式请求。

入站方向只需轻量格式包装（不是"深度转换"）：Claude Code 发来 Anthropic 格式 → 提取 system、flatten content blocks → Fusion Engine 内部用 OpenAI 格式处理 → 最终响应做反向包装回 Anthropic 格式。整个过程只是字段位置调整和名称映射，不涉及语义转换。

排除 Codex v0.81+（Responses API，国产模型不支持）后，主流编程智能体（Claude Code / Cursor / LangChain / Windsurf）的协议问题已完全消解。

4. 核心模块设计

4.1 Fusion Engine — 核心编排器

职责：接收 Gateway 转发的请求，编排完整的 Panel → Judge → Synthesizer 流水线。

方法输入输出说明
execute()OpenAI ChatRequestOpenAI ChatResponse主入口，编排完整流水线
_clone_request()ChatRequest, model_configsList[ModelRequest]按各模型适配参数
_dispatch_parallel()List[ModelRequest]List[ModelResponse]asyncio.gather + 超时/重试
_judge()原始Prompt + 三方回复JudgeAnalysis调用 Judge 模型输出结构化分析
_synthesize()JudgeAnalysis + 三方回复ChatResponse融合生成最终答案
_fallback()ChatRequestChatResponse故障降级：单模型直连

4.2 Model Adapter — 统一适配层

设计模式：策略模式 + 工厂模式。所有适配器实现统一接口 BaseAdapter，通过 AdapterRegistry 动态注册和获取。

class BaseAdapter(ABC):
    @abstractmethod
    async def chat(self, request: ModelRequest) -> ModelResponse: ...

    @abstractmethod
    async def stream_chat(self, request: ModelRequest) -> AsyncGenerator: ...

    @abstractmethod
    def adapt_request(self, openai_request: dict) -> dict:
        """将 OpenAI 格式转换为目标模型格式"""
        ...

    @abstractmethod
    def adapt_response(self, native_response: dict) -> dict:
        """将目标模型响应转回 OpenAI 格式"""
        ...
适配器API 端点格式特殊处理
DeepSeekAdapterapi.deepseek.com/v1OpenAI 原生兼容支持 reasoning_content 分离
MiniMaxAdapterapi.minimax.chat/v1OpenAI 兼容长上下文截断策略
GLMAdapteropen.bigmodel.cn/api/paas/v4OpenAI 兼容多模态 content 数组适配

4.3 Judge Module — 交叉分析器

核心 Prompt 设计：Judge 不是简单投票，而是结构化分析——找共识、找矛盾、找盲区、提取独特洞察。

你是一个严格的交叉分析裁判。你将收到同一问题的多个模型回答。请分析： 1. 共识（Consensus）：多数或全部模型一致的观点 2. 矛盾（Contradictions）：模型间的分歧点及各方立场 3. 覆盖盲区（Coverage Gaps）：所有模型都遗漏的要点 4. 独特洞察（Unique Insights）：仅某个模型提出的有价值观点 5. 置信度评估（Confidence）：对共识部分的置信度评分 输出严格 JSON 格式。

4.4 Synthesizer — 最终融合器

核心逻辑：不是简单拼接，而是基于 Judge 的分析报告，选择最优方案、补充盲区、融合独特洞察，生成连贯的最终答案。

SYNTHESIZER_PROMPT = """
你是一个综合合成器。你将收到：
1. 原始用户问题
2. 三个模型的原始回答
3. Judge 的交叉分析报告

请基于分析报告，生成一个融合后的最终答案：
- 优先采纳共识部分
- 对矛盾点给出最优判断和理由
- 补充覆盖盲区
- 整合各模型的独特洞察
- 保持回答连贯、自然、完整
"""

6.2 Fusion 策略配置 (fusion.yaml)

# Fusion 策略配置
strategies:
  default:
    panel:
      - "deepseek-v4-pro"
      - "minimax-abab7"
      - "glm-4-plus"
    judge: "minimax-abab7"
    synthesizer: "deepseek-v4-pro"
    timeout_seconds: 30
    retry_count: 2
    fallback_model: "deepseek-v4-pro"

  # 代码场景：侧重 MiniMax + DeepSeek
  code:
    panel:
      - "deepseek-v4-pro"
      - "minimax-abab7"
    judge: "deepseek-v4-pro"
    synthesizer: "minimax-abab7"

  # 推理场景：双 DeepSeek 自融合
  reasoning:
    panel:
      - "deepseek-v4-pro"
      - "deepseek-v4-pro"
    judge: "minimax-abab7"
    synthesizer: "deepseek-v4-pro"

  # 预算模式：仅用 GLM + MiniMax
  budget:
    panel:
      - "minimax-abab7"
      - "glm-4-plus"
    judge: "glm-4-plus"
    synthesizer: "minimax-abab7"

# 路由规则：根据请求特征自动选择策略
routing:
  - condition:
      content_contains: ["```", "def ", "function ", "class "]
    strategy: "code"
  - condition:
      content_contains: ["证明", "推导", "计算", "求解"]
    strategy: "reasoning"
  - condition:
      max_messages: 3
      max_tokens_estimate: 500
    strategy: "passthrough"      # 简单问题直连，省成本
  - default: true
    strategy: "default"

8. 安全与审计

🔐 PII 脱敏
  • 请求中手机号/身份证/邮箱/API Key 自动掩码
  • 发送给上游模型前脱敏，响应中不还原
  • 正则 + NER 双重检测
📊 全量审计
  • 每次 Fusion 调用全链路记录
  • 含 Panel 原始回复、Judge 分析、Synthesizer 输出
  • Redis 存储 30 天，可导出
🛡️ Prompt 注入检测
  • 检测 tool_result 中的间接注入
  • 阈值可配置，超阈值标记/拦截
  • 参考 agentfw 的 DETECT 管道
💰 成本控制
  • 实时追踪每次 Fusion 总成本
  • 按日/周/月预算上限
  • 超预算自动降级为单模型
